# 野外露营点与生态观察记录平台详细设计

## 1. 项目目标
构建一个面向露营点运营方、普通露营用户、生态观察员的综合在线平台，实现以下核心能力：

1. 露营点管理：站点资料、容量、价格、状态、地理坐标、生态指数统一维护。
2. 地图查询：支持按城市、关键字、状态、日期范围查看露营点并显示空间位置。
3. 在线预订：完成营位容量校验、日期校验、联系人信息登记与预订单生成。
4. 防刷单管控：识别重复占位、用户高频请求、IP 爆发、周末高峰异常、多帐篷高额订单。
5. 支付结算：基于预订单发起模拟支付，并支持管理员完成结算。
6. 生态观察记录：记录物种、分类、数量、时间、坐标、环境评分、栖息地等信息。
7. 数据可视化：汇总站点、订单、营收、生态分类、风控告警与空间态势。

## 2. 用户角色
### 2.1 管理员 `ADMIN`
- 管理露营点
- 查看全部预订、支付、观察记录
- 驳回或完成预订
- 审核生态记录
- 完成支付结算
- 查看风控告警与驾驶舱

### 2.2 露营用户 `USER`
- 查询露营点地图
- 创建并取消自己的预订
- 发起自己的订单支付
- 提交生态观察记录
- 查看仪表盘和自己的业务数据

### 2.3 生态观察员 `OBSERVER`
- 查询露营点地图
- 创建预订
- 发起支付
- 提交生态观察记录
- 观察记录默认直接进入已核验状态

## 3. 技术架构
### 3.1 后端
- 技术栈：Java 17、Spring Boot 3.4.3、Spring Web、Spring Data JPA、MySQL
- 目录：`server`
- 核心分层：
  - `controller`：REST 接口层
  - `service`：业务编排与规则判断
  - `repository`：JPA 数据访问
  - `entity`：领域实体
  - `config`：拦截器、跨域、种子数据
  - `common`：统一响应、异常、鉴权上下文
  - `dto`：前后端接口契约

### 3.2 前端
- 技术栈：Vite、React JSX、Ant Design、MobX、Axios、ECharts、React Leaflet
- 目录：`frontend`
- 核心结构：
  - `stores`：MobX 全局状态
  - `pages`：登录页、驾驶舱、地图页、预订页、支付页、生态记录页
  - `components`：应用外壳与布局

- 用户：`root`

- 数据库名：`camping_system`

## 4. 核心业务流程
### 4.1 登录流程
1. 用户输入用户名和密码。
2. 后端校验用户是否存在、是否启用、密码哈希是否匹配。
3. 生成基于 HMAC-SHA256 的简易登录令牌。
4. 前端保存令牌到 `localStorage`，随后拉取个人信息和全量业务数据。

### 4.2 预订流程
1. 用户选择露营点与日期范围。
2. 后端校验站点状态是否开放、日期是否合法、容量是否充足。
3. 计算订单金额：`基础价格 × 帐篷数量 × 入住天数`。
4. 执行防刷单策略。
5. 低风险订单直接创建为 `PENDING_PAYMENT`。
6. 中风险订单创建成功但打上风险标签并记录告警。
7. 高风险订单直接拦截并记录风控事件。

### 4.3 支付结算流程
1. 用户从待支付预订中选择订单。
2. 选择支付方式：微信、支付宝、银行卡。
3. 后端创建或更新支付单，状态置为 `PAID`。
4. 对应预订单状态更新为 `CONFIRMED`。
5. 管理员可对 `PAID` 状态订单执行结算，状态更新为 `SETTLED`。

### 4.4 生态记录流程
1. 用户或观察员选择露营点并填写物种信息。
2. 记录观察时间、数量、分类、栖息地、环境评分、备注等。
3. 观察员与管理员提交后默认 `VERIFIED`，普通用户提交为 `SUBMITTED`。
4. 管理员可对待审核记录执行审核通过。

## 5. 防刷单详细策略
平台在 `FraudControlService` 中实现了四类核心规则：

### 5.1 重复占位拦截
- 同一用户
- 同一露营点
- 同一日期区间存在重叠
- 且已有有效预订状态 `PENDING_PAYMENT / CONFIRMED`
- 处理动作：直接拦截，记录 `RESERVATION_BLOCK`

### 5.2 用户短时高频拦截
- 统计最近 10 分钟用户发起的预订与风控事件次数
- 大于等于 3 次视为高风险
- 处理动作：直接拦截

### 5.3 IP 爆发拦截
- 统计最近 10 分钟同源 IP 的预订与风控事件次数
- 大于等于 5 次视为高风险
- 处理动作：直接拦截

### 5.4 中风险标记
- 大团体：同行人数大于等于 8
- 多帐篷：帐篷数量大于等于 4
- 高金额：订单金额大于 2500
- 周末高峰：周末且帐篷数量大于等于 3
- 处理动作：允许下单，但标记 `MEDIUM` 风险并进入重点监测

## 7. 接口设计
### 7.1 认证接口
- `POST /api/auth/login`：登录
- `GET /api/auth/me`：获取当前用户

### 7.2 露营点接口
- `GET /api/sites`：查询露营点与地图数据
- `POST /api/sites`：创建露营点
- `PUT /api/sites/{id}`：更新露营点
- `PATCH /api/sites/{id}/status`：修改状态

### 7.3 预订接口
- `GET /api/reservations`：查询预订
- `POST /api/reservations`：创建预订
- `PATCH /api/reservations/{id}/cancel`：取消预订
- `PATCH /api/reservations/{id}/status`：管理员修改状态

### 7.4 支付接口
- `GET /api/payments`：查询支付单
- `POST /api/payments/pay`：发起支付
- `PATCH /api/payments/{id}/settle`：管理员结算

### 7.5 生态接口
- `GET /api/observations`：查询生态记录
- `POST /api/observations`：创建生态记录
- `PATCH /api/observations/{id}/verify`：管理员审核

### 7.6 仪表盘接口
- `GET /api/dashboard`：获取驾驶舱汇总数据

## 9. 安全与非功能设计
### 9.1 鉴权
- 登录后统一使用 `Authorization: Bearer <token>`
- 拦截器保护所有 `/api/**` 接口
- 登录接口除外

### 9.2 编码与字符集
- 前后端统一 UTF-8
- MySQL 建议使用 `utf8mb4`

### 9.3 可维护性
- DTO 与实体分离
- 统一响应体 `ApiResponse`
- 统一异常处理 `GlobalExceptionHandler`
- 前端通过 MobX 统一管理用户态与业务数据
